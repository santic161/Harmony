# agentic-decision — LLM Integration Reference

Use this file as compact context when generating code against `agentic-decision`.
It reflects the public API exported by the package, including:

- `DecisionOrchestrator`
- `skills` and `skillSources`
- controlled `actions`
- `rehydrator`
- `Decision<T>.usage`
- `WhatsAppWebProvider` and `WhatsAppStubProvider`
- portable/imported Agent Skills from local folders, ZIPs, URLs, SkillHub, and ClawHub

---

## What the library does

`agentic-decision` runs multi-turn LLM decision loops that can:

- ask a real person follow-up questions over a messaging channel
- call a host-defined set of runtime actions
- pause and resume across process restarts
- finalize only when the output matches a Zod schema

The result of `startDecision()` is `Promise<Decision<T>>`.

The model does not get arbitrary tool access.
It must stay inside the strict agent action protocol:

```json
{ "action": "ask", "question": "...", "reasoning": "..." }
{ "action": "propose", "proposal": "...", "confidence": 0.82, "reasoning": "..." }
{ "action": "run_action", "actionId": "inspect_release", "input": {}, "progressMessage": "...", "reasoning": "..." }
{ "action": "finalize", "value": { "...": "..." }, "confidence": 0.98, "reasoning": "..." }
{ "action": "abort", "reason": "..." }
```

## Install

```bash
pnpm add agentic-decision
```

Add only the optional peer dependencies you actually use:

```bash
# LLM providers
pnpm add openai
pnpm add @anthropic-ai/sdk
pnpm add @google/generative-ai

# Messaging channels
pnpm add node-telegram-bot-api
pnpm add whatsapp-web.js qrcode-terminal

# Optional persistence
pnpm add ioredis
```

Requirements:

- Node.js >= 20
- ESM project (`"type": "module"`)

## Core result types

```ts
interface DecisionUsage {
  calls: number;
  promptTokens: number;
  completionTokens: number;
  totalTokens: number;
  totalUsd: number;
}

interface Decision<T> {
  status: 'finalized' | 'aborted' | 'timeout';
  value: T | null;
  confidence: number;
  reason?: string;
  turns: readonly Turn[];
  costUsd?: number;
  usage?: DecisionUsage;
  sessionId: string;
}
```

`usage` is present when the underlying LLM provider returns token accounting.

## Main class

```ts
import { DecisionOrchestrator } from 'agentic-decision';

const orchestrator = new DecisionOrchestrator(options);
await orchestrator.start();
const decision = await orchestrator.startDecision(request);
await orchestrator.stop();
```

### `OrchestratorOptions`

```ts
interface OrchestratorOptions {
  llm: LLMProvider;
  messaging: readonly MessagingProvider[];
  skills?: readonly SkillDefinition[];
  skillSources?: readonly SkillSource[];
  actions?: readonly ActionDefinition[];
  store?: StateStore;
  logger?: Logger;
  events?: TypedEmitter;
  costTracker?: CostTracker;
  sanitizer?: InputSanitizer;
  guard?: PromptGuard;
  validator?: OutputValidator;
  gate?: ConfidenceGate;
  systemPrompt?: string;
  maxTurns?: number;
  userReplyTimeoutMs?: number;
  llmTimeoutMs?: number;
  llmTemperature?: number;
  llmMaxTokens?: number;
  rehydrator?: SessionRehydrator;
}
```

Important notes:

- `skills` is the static in-memory skill registry.
- `skillSources` is a portable/imported skill registry that is resolved asynchronously during `start()`.
- imported and static skills share one merged id namespace.
- duplicate skill ids throw a clear error.
- imported skills still require explicit activation through `skillIds`.

### `StartDecisionRequest<T>`

```ts
interface StartDecisionRequest<T> {
  prompt: string;
  userId: string;
  channel: string;
  externalUserId: string;
  schema: ZodType<T>;
  metadata?: Readonly<Record<string, string | number | boolean>>;
  kickoffMessage?: string;
  skillIds?: readonly string[];
  allowedActionIds?: readonly string[];
}
```

### Blocking vs background execution

```ts
// blocking
const decision = await orchestrator.startDecision({ ... });

// background
void orchestrator.startDecision({ ... }).then((decision) => {
  console.log(decision.status);
});
```

## Quick start

```ts
import { z } from 'zod';
import OpenAI from 'openai';
import {
  ConsoleProvider,
  DecisionOrchestrator,
  OpenAIProvider,
  type OpenAILike,
} from 'agentic-decision';

const ApprovalSchema = z.object({
  approved: z.boolean(),
  summary: z.string().min(1),
});

const orchestrator = new DecisionOrchestrator({
  llm: new OpenAIProvider({
    client: new OpenAI({
      apiKey: process.env.OPENAI_API_KEY!,
    }) as unknown as OpenAILike,
    model: 'gpt-4o-mini',
  }),
  messaging: [new ConsoleProvider({ externalUserId: 'local-user' })],
  skills: [
    {
      id: 'release-review',
      description: 'Gather release evidence before recommending deployment.',
      instructions:
        'Ask focused questions, summarize risk, and finalize only after confirmation.',
    },
  ],
});

const decision = await orchestrator.startDecision({
  prompt: 'Decide whether tonight\'s deployment should proceed.',
  userId: 'release-manager',
  channel: 'console',
  externalUserId: 'local-user',
  schema: ApprovalSchema,
  skillIds: ['release-review'],
  kickoffMessage: 'Release review started. Please answer in the terminal.',
});

console.log(decision.status, decision.value, decision.usage?.totalTokens);
await orchestrator.stop();
```

## Messaging providers

All providers conform to:

```ts
interface MessagingProvider {
  readonly name: string;
  start(): Promise<void>;
  stop(): Promise<void>;
  send(to: string, msg: OutboundMessage): Promise<void>;
  onReply(handler: InboundHandler): () => void;
}
```

Bundled providers:

- `ConsoleProvider`
- `TelegramProvider`
- `WhatsAppWebProvider`
- `WhatsAppStubProvider`

### `WhatsAppWebProvider`

Use this when you have a real `whatsapp-web.js`-compatible client:

```ts
import { Client, LocalAuth } from 'whatsapp-web.js';
import { WhatsAppWebProvider } from 'agentic-decision';

const client = new Client({ authStrategy: new LocalAuth() });

const whatsapp = new WhatsAppWebProvider({
  client,
  initializeClientOnStart: true,
  destroyClientOnStop: false,
});
```

### `WhatsAppStubProvider`

Use this when you want a light adapter for tests, demos, or your own message bridge:

```ts
import { WhatsAppStubProvider } from 'agentic-decision';

const whatsapp = new WhatsAppStubProvider({
  sendHook: async (to, msg) => {
    console.log(`[stub -> ${to}] ${msg.text}`);
  },
});

await whatsapp.start();
whatsapp.inject({
  externalUserId: '15551234567@c.us',
  text: 'yes, confirm',
});
```

## Controlled actions

Actions are registered globally and enabled per decision through `allowedActionIds`.

### Handler action

```ts
{
  id: 'inspect_release',
  title: 'Inspect release',
  description: 'Returns trusted deployment evidence from the host runtime.',
  kind: 'handler',
  inputSchema: z.object({}),
  execute: async () => ({
    risk: 'low',
    rollbackReady: true,
  }),
}
```

### Shell action

```ts
{
  id: 'run_focused_tests',
  title: 'Run focused tests',
  description: 'Runs the targeted Vitest suite.',
  kind: 'shell',
  inputSchema: z.object({}),
  command: process.platform === 'win32' ? 'pnpm.cmd' : 'pnpm',
  buildArgs: () => ['exec', 'vitest', 'run', 'test/AgentEngine.test.ts'],
  cwd: process.cwd(),
  envAllowlist: ['CI'],
}
```

Security model for actions:

- the model never emits raw shell text
- action ids must be registered by the host
- action input is Zod-validated before execution
- shell actions use fixed commands plus validated args

## Static skills

Static skills are plain objects:

```ts
interface SkillDefinition {
  id: string;
  description: string;
  instructions: string;
  preferredActionIds?: readonly string[];
  examples?: readonly string[];
  source?: ImportedSkillSource;
  resources?: readonly SkillResource[];
  license?: string;
  compatibility?: string;
  metadata?: Readonly<Record<string, unknown>>;
  allowedTools?: readonly string[];
  diagnostics?: readonly string[];
}
```

Example:

```ts
const skills = [
  {
    id: 'repo-audit',
    description: 'Inspect the repository and summarize evidence before deciding.',
    instructions:
      'Prefer repository inspection first, then summarize risk and ask for confirmation.',
    preferredActionIds: ['inspect_files', 'read_package_json'],
  },
];
```

Only skills listed in `skillIds` are injected into a given decision.

## Portable / imported Agent Skills

Portable skills are imported `SKILL.md` bundles.

The library exposes these helpers:

```ts
import {
  loadSkillFromDirectory,
  loadSkillFromZip,
  loadSkillFromUrl,
  loadSkillFromClawHub,
  writeImportedSkill,
  SkillHubRegistryClient,
} from 'agentic-decision';
```

### Portable skill types

```ts
interface ImportedSkill {
  id: string;
  description: string;
  instructions: string;
  source: ImportedSkillSource;
  resources: readonly SkillResource[];
  license?: string;
  compatibility?: string;
  metadata?: Readonly<Record<string, unknown>>;
  allowedTools?: readonly string[];
  diagnostics?: readonly string[];
}

type SkillSource =
  | { kind: 'directory'; path: string }
  | { kind: 'zip'; path: string }
  | { kind: 'url'; url: string; headers?: Readonly<Record<string, string>>; timeoutMs?: number }
  | { kind: 'clawhub'; url: string; headers?: Readonly<Record<string, string>>; timeoutMs?: number };
```

### What is parsed from `SKILL.md`

The importer reads:

- YAML frontmatter
- Markdown body

Mappings:

- `name` -> `id` (normalized)
- `description` -> `description`
- Markdown body -> `instructions`

Preserved optional metadata:

- `license`
- `compatibility`
- `metadata`
- `allowed-tools` -> `allowedTools`

The parser is intentionally lenient for compatibility:

- it tolerates common YAML mistakes when possible
- it emits diagnostics when it had to recover
- it throws only when the file is irrecoverable or no usable description exists

### Local directory

```ts
const imported = await loadSkillFromDirectory('/absolute/path/to/portable-skill');
console.log(imported.id, imported.resources);
```

### Local ZIP

```ts
const imported = await loadSkillFromZip('/tmp/release-review.zip');
```

### Generic remote URL

The URL may point directly to:

- a raw `SKILL.md`
- a ZIP bundle
- a supported page that embeds portable skill content

```ts
const imported = await loadSkillFromUrl('https://example.com/skills/release-review.zip');
```

### ClawHub import

ClawHub is supported for explicit page/ZIP import.
In v1, the loader resolves the page download link in a controlled way.

```ts
const imported = await loadSkillFromClawHub(
  'https://clawhub.ai/skills/release-review',
);
```

### SkillHub search and import

SkillHub catalog access uses an explicit API key.

```ts
const registry = new SkillHubRegistryClient({
  apiKey: process.env.SKILLHUB_API_KEY!,
});

const matches = await registry.search({ query: 'release review', limit: 5 });
const catalog = await registry.browse({ limit: 20, sort: 'recent' });

const first = matches[0];
if (first?.pageUrl) {
  const imported = await loadSkillFromUrl(first.pageUrl);
  console.log(imported.id);
}
```

### Persist an imported skill explicitly

The library never writes imported skills automatically.
If the host wants to persist one, it must choose the target directory explicitly.

```ts
const imported = await loadSkillFromUrl('https://example.com/skills/release-review.zip');
await writeImportedSkill(imported, '/explicit/target/directory');
```

### Use imported skills in the orchestrator

```ts
const orchestrator = new DecisionOrchestrator({
  llm,
  messaging: [consoleProvider],
  skillSources: [
    { kind: 'directory', path: '/skills/release-review' },
    { kind: 'clawhub', url: 'https://clawhub.ai/skills/release-review' },
  ],
});

const decision = await orchestrator.startDecision({
  prompt: 'Review tonight\'s deployment.',
  userId: 'release-manager',
  channel: 'console',
  externalUserId: 'local-user',
  schema: ApprovalSchema,
  skillIds: ['portable-release-review'],
});
```

Runtime behavior:

- `DecisionOrchestrator.start()` resolves `skillSources` asynchronously
- imported skills are merged with static `skills`
- duplicate ids fail deterministically
- imported skills participate in the existing `skillIds` activation flow
- only active skills are injected into the prompt
- imported skills add compact `source`, `compatibility`, and resource inventory context to the prompt

### Security policy for imported skills

This is the intended v1 safety model:

- remote imports are opt-in only
- the runtime never downloads skills based on prompt matching
- bundled `scripts/`, `references/`, and `assets/` are inventory, not implicit permissions
- `allowed-tools` is informational metadata only
- imported `allowedTools` is not auto-mapped to `allowedActionIds`
- no remote scripts or install commands are executed automatically
- ZIP paths are sanitized to block traversal

## Persistence and rehydration

Bundled stores:

- `InMemoryStateStore`
- `RedisStateStore`

Use `rehydrator` when an active session may survive a restart:

```ts
const orchestrator = new DecisionOrchestrator({
  llm,
  messaging: [consoleProvider],
  store: redisStore,
  rehydrator: async (session) => {
    if (session.metadata?.workflow === 'release-review') {
      return { schema: ApprovalSchema };
    }
    return null;
  },
});
```

Behavior:

- stored active sessions are reloaded during `start()`
- the `rehydrator` decides which schema to reattach
- if rehydration is impossible, the stale session is aborted instead of staying locked forever

## Reliability, validation, and safety

Built-in exports include:

- `ConfidenceGate`
- `CostTracker`
- `retry`
- `withTimeout`
- `CircuitBreaker`
- `InputSanitizer`
- `PromptGuard`
- `OutputValidator`
- `sanitizeErrorMessage`

Default behavior:

- invalid model JSON triggers a repair loop
- final payloads must pass your Zod schema
- user text is wrapped in `<user_input>...</user_input>`
- low-confidence proposals require follow-up or confirmation
- sessions time out if the user never replies
